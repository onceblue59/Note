# iNote 聊天笔记维护协议

本文件是 `/iNote` 命令的唯一仓库执行规则。

目标：在 GPT 网页聊天中，将当前对话里与用户要求相关的内容整理为可独立阅读、可长期维护的个人笔记，并通过 GitHub 工具写入本仓库。

默认执行目标：

```text
onceblue59/Note@main
```

除非用户明确指定，否则禁止切换仓库或分支。不得使用旧规则、记忆或其他仓库中的同名文件解释 `/iNote`。

## 1. 触发与输入

只有当用户消息以 `/iNote` 开头时，才执行本协议。

`/iNote` 后面的全部文字是本次操作要求，不是笔记正文。

待处理材料包括：

1. 当前聊天中与本次要求相关的用户消息和助手回复。
2. `/iNote` 后指定的范围、重点、标题、栏目、标签、篇幅、风格及新增或更新要求。
3. 仓库中与主题相关的已有文章。

不得把 `/iNote` 命令本身写入正文。

## 2. 执行原则

1. 修改仓库前必须先读取本文件。
2. 本文件引用其他仓库文件时，继续读取相关文件。
3. 先解析要求和选择上下文，再搜索已有文章。
4. 先决定新增或更新，再生成最终内容。
5. 更新文章前必须读取目标文件完整内容。
6. 只修改本文件授权的文件。
7. 不把聊天逐轮复制为文章。
8. 不引入聊天或仓库中没有依据的新事实。
9. 非关键参数缺失时按默认规则推断，不反复追问。
10. 工具未实际成功前，不得声称文件已读取、修改或提交。
11. 任一步骤失败时，停止依赖该步骤的后续操作，并说明失败原因。

## 3. 参数解释

`/iNote` 后使用自然语言表达，不要求固定命令语法。

可指定：

- 整理范围、主题、标题、栏目、标签
- 重点内容、排除内容、篇幅、结构、风格
- 是否保留推理过程、反例或未解决问题
- 新增、更新、追加、合并
- 目标文件名或文章标题
- 草稿、精选状态

决策优先级：

1. 当前 `/iNote` 命令中的明确要求。
2. 当前聊天中用户此前明确表达的要求。
3. 用户指定的已有目标文章及其结构。
4. 仓库现有内容。
5. 本文件的默认规则。

低优先级规则不得覆盖高优先级要求。

若明确要求互相矛盾且无法按优先级解决，立即停止，不写入仓库。

## 4. 上下文选择

默认选择当前对话中与请求主题相关的连续内容，通常从最近一次明显的主题起点开始，到 `/iNote` 命令之前结束。

如果聊天包含多个无关主题，只选择直接相关部分，不机械处理整段会话。

用户明确指定起点、终点、最近几轮、整段对话、主题或排除范围时，优先遵循。

内容价值优先级：

1. 用户提出的问题、判断和修正。
2. 核心概念、定义、区别和因果关系。
3. 可执行的方法、流程、标准和检查方式。
4. 重要边界、反例和未解决问题。
5. 助手提供的有效补充。
6. 寒暄、确认、重复说明和过程性话语。

如果没有足够内容形成有效笔记，立即停止。不得创建空文章或依靠模型常识凭空扩写。

## 5. 聊天转笔记

输出必须是可独立阅读的个人笔记，不是聊天摘要或聊天记录。

整理时必须：

- 删除寒暄、确认语、工具说明、助手套话和无关内容。
- 合并重复观点。
- 修正口语、省略和指代不清。
- 将问答顺序重组为主题结构。
- 保留关键问题、定义、判断、推理、方法、结论、边界和未解决问题。
- 必要时区分事实、推断、暂时判断和问题。
- 保持原意，不把助手意见写成用户已经接受的观点。

默认不使用"用户问""助手回答""在本次对话中""希望对你有帮助"等聊天或助手式表达。

不得：

- 添加没有依据的新结论、案例、引用、数据或来源。
- 虚构用户经历。
- 把暂时讨论写成确定事实。
- 删除影响结论成立的重要条件。
- 默认声明文章由 AI 或 ChatGPT 生成。

文章语气应直接、克制，优先保留定义、区分、因果关系、判断标准和执行方法。

## 6. 正文结构

根据内容选择结构，不生成空章节。

复杂主题可使用：

```md
# 标题

## 问题背景
## 核心判断
## 关键概念
## 推理过程
## 执行方法
## 边界与反例
## 尚未解决的问题
```

较短内容可使用：

```md
# 标题

## 核心观点
## 具体方法
```

若原内容有更合适的结构，应采用更自然的结构。

## 7. 新增、更新与合并

默认新增文章。

以下情况优先更新已有文章：

- 用户明确要求更新、追加、补充或合并。
- 用户指定已有文件名或文章标题。
- 当前讨论明显是已有笔记的延续。
- 存在主题高度一致且适合作为自然扩展的文章。

如果无法可靠判断目标文件，默认新增，不得覆盖可能无关的文章。

更新时：

1. 读取目标文件完整内容。
2. 保留原 `pubDate`。
3. 将 `updatedDate` 更新为当前日期。
4. 保留仍有效的原有内容。
5. 将新内容整合进合适章节，不简单追加"本次对话总结"。
6. 除非用户明确要求，不修改文件名和 URL。
7. 新内容修正旧判断时，应保留修正关系。

合并时：

- 读取所有指定源文章。
- 选择一个目标文件并消除重复。
- 重组为统一结构。
- 未经明确授权，不删除源文件。

## 8. 仓库结构与授权范围

本项目是 Astro 内容站点。

主要文件：

- 文章目录：`src/content/posts/`
- 文章 schema：`src/content/config.ts`
- 栏目数据：`src/data/categories.ts`
- 栏目图片：`public/images/covers/`

正常新增或更新文章时，只允许修改：

```text
src/content/posts/<slug>.md
```

新增栏目时才允许修改：

```text
src/content/config.ts
src/data/categories.ts
public/images/covers/<slug>.svg
```

不得因普通笔记操作修改页面模板、路由、RSS、样式、依赖、构建配置或本规则文件。

## 9. 文章文件

文件位于 `src/content/posts/`。

文件名：

- 使用小写英文、数字和连字符。
- 不使用中文、空格和特殊符号。
- 默认扩展名为 `.md`。
- 文件名即文章 URL slug。

根据标题生成简短、稳定、可识别的英文 slug。中文标题需翻译或概括。

优先生成语义明确的文件名，例如：

```text
agent-core-process.md
agent-context-management.md
```

不要优先生成 `note-2.md` 一类机械名称。只有无法生成更明确且不冲突的 slug 时才追加数字后缀。

新增前必须检查相同文件名、相同或近似标题以及高度相关主题，避免无意义重复。

## 10. Frontmatter

新文章模板：

```md
---
title: "文章标题"
description: "一句话说明文章内容，建议 30 到 80 字。"
pubDate: 2026-07-14
updatedDate: 2026-07-14
category: "study-notes"
tags: ["标签一", "标签二"]
featured: false
heroImage: "images/covers/study.svg"
draft: false
---

正文
```

字段规则：

| 字段 | 规则 |
| --- | --- |
| `title` | 必填，清楚表达具体主题。 |
| `description` | 必填，约 30 到 80 字，概括主题与价值。 |
| `pubDate` | 必填；新增使用当前日期，更新时保留原值。 |
| `updatedDate` | 新增可与 `pubDate` 相同；更新使用当前日期。 |
| `category` | 必填；必须同时存在于 `src/content/config.ts` 和 `src/data/categories.ts`。 |
| `tags` | 建议 1 到 4 个关键词，默认 `[]`。 |
| `featured` | 默认 `false`；用户明确要求时才设为 `true`。 |
| `heroImage` | 从 `public/` 下一级开始写，不含 `public/`，不以 `/` 开头。 |
| `draft` | 默认 `false`；用户说草稿、暂不发布或先保存时设为 `true`。 |

标题优先使用用户指定内容；否则根据核心问题和结论概括。避免"聊天总结""一些思考"等空泛标题。

标签应保持同一概念同一写法，不机械重复栏目名，不使用无区分价值的宽泛词。

设置 `featured: true` 前应检查已有精选文章。未经授权不得取消旧精选；若出现多篇精选，应在结果中说明。

图片示例：

```text
public/images/covers/study.svg
```

frontmatter 中写：

```md
heroImage: "images/covers/study.svg"
```

## 11. 当前栏目

优先使用已有栏目：

| slug | 栏目名 | 适合内容 |
| --- | --- | --- |
| `private-menu` | 私房菜单 | 菜谱、备菜、饮食和餐桌经验 |
| `study-notes` | 学习笔记 | 课程、教程、论文和技术概念 |
| `reading-notes` | 阅读笔记 | 摘要、章节理解和阅读追问 |
| `essays` | 随笔 | 生活片段、短文和未系统化想法 |
| `thinking` | 思考 | 判断、困惑、推演和反思 |
| `professional-learning` | 专业学习 | 能力栈、工程实践、方法论和学习路线 |
| `projects` | 项目记录 | 项目背景、实现、取舍、踩坑和复盘 |
| `tools` | 工具方法 | 命令行、编辑器、自动化和信息管理 |
| `life` | 生活观察 | 城市、关系、身体、节奏和日常经验 |
| `media` | 书影音 | 电影、音乐、播客、展览和作品短评 |

用户未指定栏目时，根据主题选择。无法可靠判断时使用 `essays`。

不得仅为一篇文章临时新增栏目，除非现有栏目确实无法承载且用户要求或明确同意新增。

## 12. 新增栏目

新增栏目必须同时修改：

1. `src/content/config.ts`
2. `src/data/categories.ts`

按需新增：

```text
public/images/covers/<slug>.svg
```

slug 使用小写英文和连字符，发布后尽量不修改。

`config.ts` 中的 slug 与 `categories.ts` 中的 `slug` 必须完全一致。

栏目对象包含：

```ts
{
  slug: 'travel-notes',
  name: '旅行笔记',
  deck: '路线、地方与抵达之后的观察',
  description: '记录旅行路线、城市观察、途中经验和可复用的行前准备。',
  accent: '#4f7c9f',
  image: 'images/covers/travel.svg'
}
```

如无新图，可复用已有封面，但必须确认目标文件存在。

## 13. GitHub 执行流程

每次 `/iNote` 按以下顺序执行：

1. 确认目标为 `onceblue59/Note@main`。
2. 读取 `.agents/iNote.md`。
3. 解析 `/iNote` 后的全部要求。
4. 选择相关聊天上下文。
5. 读取 `src/content/config.ts` 与 `src/data/categories.ts`。
6. 搜索相关已有文章并按需读取完整内容。
7. 决定新增、更新或合并。
8. 生成最终 Markdown。
9. 检查目标路径与授权范围。
10. 使用 GitHub 工具写入 `main`。
11. 重新读取目标文件，验证远端内容已经更新。
12. 返回实际结果。

GitHub Contents API 的成功写入本身会生成提交，因此不得再声称存在额外未执行的"推送"步骤。

默认提交信息：

```text
content: add <slug>
content: update <slug>
content: add category <slug>
```

每次操作只提交本次授权的文件。

## 14. 构建验证

仓库构建命令为：

```sh
pnpm run build
```

它用于执行 Astro 检查与构建。

只有在当前执行环境确实提供代码检出和命令执行能力时，才运行构建并声明构建结果。

如果当前仅有 GitHub 文件 API、无法运行 shell：

- 不得声称构建通过。
- 文件写入成功后，应明确说明"远端提交已完成，但本次环境未执行构建"。
- 不得因无法运行 shell 而虚构验证结果。

若实际运行构建且失败，不得声称任务全部完成，应说明提交状态和构建错误。

## 15. 完成检查

新增或更新文章后检查：

1. 文件位于 `src/content/posts/`。
2. 文件名符合 slug 规则。
3. frontmatter 包含必填字段。
4. `category` 是已登记 slug。
5. `heroImage` 不含 `public/` 且目标文件存在。
6. `draft`、`featured` 符合用户要求。
7. 正文不是聊天逐字稿。
8. 未加入无依据的新事实。
9. 远端文件重新读取后内容正确。

新增栏目后检查：

1. 两个栏目定义文件均包含新 slug。
2. slug 拼写完全一致。
3. 图片存在或明确复用已有图片。
4. 至少有一篇文章使用该栏目，除非用户明确只预建栏目。

## 16. 停止条件

出现以下情况时立即停止，不继续写入：

- 无法读取本规则文件。
- `/iNote` 参数无法解析。
- 当前聊天没有足够有效内容。
- 明确要求相互冲突且无法解决。
- 无法确定合法栏目且默认 `essays` 也不适用。
- 用户要求修改未授权文件。
- 无法可靠识别更新目标。
- GitHub 读取或写入失败。
- 目标文件在读取后发生版本冲突。

停止时说明具体失败步骤和原因，不提供虚假的成功结论。

## 17. 返回格式

成功后简洁返回：

- 操作：新增、更新或合并
- 标题
- 文件路径
- 页面 URL
- 栏目
- 标签
- 草稿或精选状态（如适用）
- Git 提交 SHA
- 构建状态：已通过、失败或本环境未执行

默认不重复完整文章正文，除非用户明确要求预览或查看全文。

部分成功时必须分别说明：

- 哪些文件已经实际提交
- 哪些步骤未执行或失败
- 是否执行了构建

## 18. 常见错误

错误栏目：

```md
category: "study-note"
```

正确：

```md
category: "study-notes"
```

错误图片路径：

```md
heroImage: "public/images/covers/study.svg"
```

正确：

```md
heroImage: "images/covers/study.svg"
```

只修改一个栏目定义文件会导致 schema 与页面元数据不一致。

文件写入成功不等于构建已经通过；只有实际运行构建后才能报告构建结果。
